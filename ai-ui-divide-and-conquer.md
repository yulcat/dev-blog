# AI와 함께 게임 UI 빠르게 만들기: 선언에서 시작하는 분할정복

> 초안 작성 중

## 핵심 주장

선언(protobuf, SerializeField)을 진실의 원천으로 삼으면, AI가 요구사항을 "상상"할 여지가 없어진다. 그 결과 hallucination 없이 빠른 자동완성이 가능하다.

---

## 용어 정리

- **데이터 protobuf** — 서버팀이 제공하는 데이터 구조 선언. 클라 작업 전에 미리 주어짐.
- **API protobuf** — 서버팀이 제공하는 API 요청/응답 구조 선언. 클라 작업 전에 미리 주어짐.
- **SerializeField** — UI 작업자가 제공한 프리팹을 보고 클라 작업자가 직접 작성하는 Unity 직렬화 필드 선언.
- **레코드(Record)** — 클라이언트 캐시 레이어. 서버 데이터를 클라이언트에서 보관/관리하는 단위.
- **UI View** — SerializeField 기반으로 생성되는 UI 표현 코드.
- **프레젠터(Presenter)** — VContainer를 사용해 데이터 테이블, 레코드, API, View를 주입받고 실제 동작을 정의하는 레이어. 예: View 이벤트 수신 → 데이터 조건 체크 → API 호출 → 레코드 업데이트 → View 갱신.

---

## 구조 다이어그램

```
[주어지는 것]
  데이터 protobuf
  API protobuf
  SerializeField (프리팹 기반)

         ↓ 병렬 생성 가능

  데이터 코드 ──┐
  레코드 코드 ──┼──→ API 코드 (데이터+레코드 필요)
  UI View 코드 ─┘         ↓
                      프레젠터 (VContainer로 전체 주입 및 연결)
```

## 1. 들어가며

AI 코딩 도구를 써보고 싶은데, 막상 게임 클라이언트 UI 작업에 어떻게 적용해야 할지 막막한 경우가 많다. "어디서부터 시켜야 하지?", "이걸 기대해도 되나?" — 이 글은 그 질문에 답한다.

핵심은 이것이다. **AI는 요구사항의 60% 이상이 구체화되어 있으면 코드를 자동완성할 수 있다.** 여기서 "자동완성"이라는 비유가 중요하다. IDE의 자동완성이 앞뒤 맥락을 보고 나머지를 채우듯, AI도 충분한 맥락이 주어지면 나머지를 정확하게 채운다. 맥락이 부족하면 AI는 빈 자리를 상상으로 채우고, 그게 문제의 시작이다.

그렇다면 전략은 명확하다. **자동완성된 코드를 다음 자동완성의 재료로 쓰는 것.** 뿌요뿌요에서 하나의 연쇄가 다음 연쇄를 만들어내듯, 구체화된 코드 조각이 쌓이면 더 큰 구현을 자동완성할 수 있게 된다.

이 글에서는 게임 클라이언트 UI 개발에서 이 연쇄를 어떻게 설계하고 실제로 적용했는지를 소개한다.
